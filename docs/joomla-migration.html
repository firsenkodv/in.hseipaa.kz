Миграция пользователей с Joomla на Laravel

Документация по кастомному UserProvider для плавной миграции учётных записей

Обзор задачи

Сайт переехал с Joomla на Laravel 12. В базе данных осталась старая таблица pc0ec_users со всеми учётными записями пользователей из Joomla. Стандартная Laravel-авторизация работает с таблицей users, которая изначально пуста.

Цель: Реализовать «плавную» миграцию — пользователи продолжают логиниться со своими старыми паролями. При первом успешном входе запись автоматически переносится из pc0ec_users в users, а исходная запись удаляется. Таким образом, в pc0ec_users в любой момент видно, кто ещё не зашёл на новый сайт.

✅

Пользователь не замечает миграции — он просто вводит свой старый пароль и оказывается залогинен. Никакого сброса паролей и писем не требуется.

Проблема хэшей паролей

Laravel хранит пароли в формате bcrypt (строки вида $2y$10$...). Joomla исторически использовала несколько форматов — в зависимости от версии:

Формат Пример Встречается в
Чистый MD5 7c83664d540b0424afd3a25e5ecdbf0f Joomla 1.x — 2.x
bcrypt $2y$10$9hB.erUcl7udOzSq7MvjGe... Joomla 3.2+

В данном проекте в таблице pc0ec_users обнаружены оба формата — сайт Joomla обновлялся, поэтому старые пользователи остались с MD5, а новые получили bcrypt.

⚠️

Почему нельзя просто перенести хэши?
Хэширование — необратимая операция. Нельзя взять MD5-хэш и превратить его в bcrypt-хэш того же пароля, не зная исходного пароля. Единственный момент, когда исходный пароль доступен — это момент входа пользователя.

Архитектура решения

Laravel использует концепцию UserProvider — объекта, который отвечает за поиск пользователя в хранилище и проверку его пароля. Стандартный провайдер — EloquentUserProvider — работает только с одной таблицей и только с bcrypt. Мы создаём свой провайдер, расширяя стандартный.

Задействованные файлы

Файл Роль
app/Auth/JoomlaUserProvider.php Основная логика: поиск в двух таблицах, проверка MD5/bcrypt, миграция
app/Providers/AppServiceProvider.php Регистрация нового провайдера в Laravel под именем joomla

Файл 1 — JoomlaUserProvider.php

Расположен по пути app/Auth/JoomlaUserProvider.php.

Это центральный файл всей логики. Класс наследует Illuminate\Auth\EloquentUserProvider и переопределяет два метода: retrieveByCredentials() и validateCredentials().

app/Auth/JoomlaUserProvider.php
namespace App\Auth;

use App\Models\User;
use Illuminate\Auth\EloquentUserProvider;
use Illuminate\Contracts\Auth\Authenticatable;
use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Hash;

class JoomlaUserProvider extends EloquentUserProvider
{
    private ?object $pendingJoomlaUser = null;

    public function retrieveByCredentials(array $credentials): ?Authenticatable
    {
        $user = parent::retrieveByCredentials($credentials);

        if ($user) {
            $this->pendingJoomlaUser = null;
            return $user;
        }

        $email = $credentials['email'] ?? null;
        if (!$email) {
            return null;
        }

        $joomlaUser = DB::table('pc0ec_users')->where('email', $email)->first();
        if (!$joomlaUser) {
            return null;
        }

        $this->pendingJoomlaUser = $joomlaUser;

        $placeholder = new User();
        $placeholder->forceFill([
            'email'    => $joomlaUser->email,
            'username' => $joomlaUser->username ?? $joomlaUser->name,
        ]);

        return $placeholder;
    }

    public function validateCredentials(Authenticatable $user, array $credentials): bool
    {
        if ($this->pendingJoomlaUser === null) {
            return parent::validateCredentials($user, $credentials);
        }

        $joomla        = $this->pendingJoomlaUser;
        $plainPassword = $credentials['password'];
        $hash          = $joomla->password;

        if (str_starts_with($hash, '$2y$')) {
            $valid = Hash::check($plainPassword, $hash);
        } else {
            $valid = hash_equals(md5($plainPassword), $hash);
        }

        $this->pendingJoomlaUser = null;

        if (!$valid) {
            return false;
        }

        $newUser = User::create([
            'email'      => $joomla->email,
            'username'   => $joomla->username ?? $joomla->name,
            'password'   => $plainPassword,
            'published'  => $joomla->block ? 0 : 1,
            'created_at' => $joomla->registerDate ?: now(),
        ]);

        DB::table('pc0ec_users')->where('id', $joomla->id)->delete();

        $user->setRawAttributes($newUser->getAttributes());
        $user->exists = true;

        return true;
    }
}

Свойство $pendingJoomlaUser

Приватное свойство провайдера, которое хранит объект пользователя из таблицы pc0ec_users между вызовами двух методов.

Это необходимо, потому что Laravel вызывает retrieveByCredentials() и validateCredentials() по очереди — и оба они нужны для полной картины. В первом методе у нас есть email, во втором — пароль. Свойство «передаёт» найденного Joomla-пользователя между ними.

Метод retrieveByCredentials()

Этот метод вызывается Laravel первым при каждом вызове auth()->attempt(). Его задача — найти пользователя по реквизитам (email), но без проверки пароля.

Алгоритм работы

  1. Вызываем parent::retrieveByCredentials($credentials) — это стандартный поиск в таблице users по email.
  2. Если пользователь найден в users — сбрасываем $pendingJoomlaUser в null и возвращаем модель. Всё идёт стандартным путём.
  3. Если в users не найден — берём email из массива credentials и ищем в pc0ec_users через DB::table().
  4. Если и там не найден — возвращаем null. Laravel поймёт, что пользователь не существует, и вернёт ошибку.
  5. Если найден в pc0ec_users — сохраняем всю запись в $pendingJoomlaUser и создаём заглушку: новый несохранённый объект User с заполненными email и username.

Зачем нужна «заглушка»?

Laravel ожидает, что retrieveByCredentials() вернёт объект, реализующий интерфейс Authenticatable. Именно этот объект передаётся дальше в validateCredentials() и, при успехе, используется для создания сессии. Мы создаём пустой несохранённый User, чтобы удовлетворить ожидание Laravel — а позже, в validateCredentials(), подменим его атрибуты реальными данными нового пользователя.

ℹ️

Поиск в pc0ec_users выполняется без модели Eloquent — через DB::table(). Это сделано намеренно: у Joomla-таблицы нет Laravel-модели, а создавать её нет смысла — таблица временная и со временем опустеет.

Метод validateCredentials()

Вызывается вторым. Получает объект пользователя (заглушку) и полный массив credentials, включая введённый пароль. Должен вернуть true или false.

Алгоритм работы

  1. Проверяем $pendingJoomlaUser. Если null — значит, пользователь нашёлся в стандартной таблице users, и мы делегируем проверку родительскому методу (стандартный bcrypt).
  2. Если $pendingJoomlaUser установлен — извлекаем из него хэш пароля и получаем введённый пользователем пароль из credentials.
  3. Определяем тип хэша и проверяем пароль (подробнее в разделе «Типы хэшей»).
  4. Сбрасываем $pendingJoomlaUser = null, чтобы не оставалось «мусора» в памяти провайдера.
  5. Если пароль не совпал — возвращаем false. Laravel покажет ошибку логина. Пользователь остаётся в pc0ec_users.
  6. Если пароль совпал — создаём запись в users через User::create() с rehash-ом пароля в bcrypt.
  7. Удаляем запись из pc0ec_users по её id.
  8. Вызываем $user->setRawAttributes($newUser->getAttributes()) и $user->exists = true — это подменяет атрибуты заглушки реальными данными только что созданного пользователя, включая его id. Laravel запишет этот ID в сессию и пользователь окажется залогинен корректно.

Почему важна подмена атрибутов?

После того как validateCredentials() вернул true, Laravel вызывает auth()->login($user), где $user — тот самый объект, который вернул retrieveByCredentials(). Он записывает в сессию значение $user->getAuthIdentifier(), то есть $user->id.

Если бы мы не подменили атрибуты, заглушка не имела бы id (он null), и сессия оказалась бы сломана. Поэтому мы принудительно копируем все атрибуты нового пользователя через setRawAttributes(), минуя кастеры модели.

⚠️

setRawAttributes() используется специально, а не forceFill(), чтобы не прогонять значения через кастеры Eloquent. Например, поле password имеет каст 'hashed' — при записи через setAttribute() оно было бы захэшировано повторно.

Проверка типов хэшей

В методе validateCredentials() определение типа хэша происходит по первым символам строки:

Фрагмент из validateCredentials()
if (str_starts_with($hash, '$2y$')) {
    // bcrypt — проверяем через встроенный фасад Hash
    $valid = Hash::check($plainPassword, $hash);
} else {
    // MD5 без соли — вычисляем md5 от пароля и сравниваем
    $valid = hash_equals(md5($plainPassword), $hash);
}
Хэш начинается с Алгоритм проверки Функция PHP
$2y$ bcrypt Hash::check($password, $hash)
всё остальное (32 hex-символа) MD5 hash_equals(md5($password), $hash)

Почему hash_equals, а не ===?

Функция hash_equals() сравнивает строки за постоянное время, независимо от длины совпадающего префикса. Это защита от timing attack — атаки по времени, при которой злоумышленник может угадать хэш по разнице в скорости ответа сервера. При использовании обычного === такая атака теоретически возможна.

Файл 2 — AppServiceProvider.php

Расположен по пути app/Providers/AppServiceProvider.php.

В метод boot() добавлена регистрация нового провайдера под именем 'joomla':

app/Providers/AppServiceProvider.php — метод boot()
Auth::provider('joomla', function ($app, array $config) {
    return new JoomlaUserProvider($app['hash'], $config['model']);
});

Auth::provider() — это метод Laravel, который регистрирует фабрику для создания провайдера с указанным именем. Когда Laravel видит в конфиге 'driver' => 'joomla', он вызывает эту функцию и получает готовый экземпляр JoomlaUserProvider.

Конструктор JoomlaUserProvider принимает два аргумента:

  • $app['hash'] — сервис хэширования Laravel (нужен родительскому классу для bcrypt)
  • $config['model'] — класс модели пользователя, указанный в auth.php

Файл 3 — config/auth.php

Расположен по пути config/auth.php.

Единственное изменение — в секции providers значение driver изменено с 'eloquent' на 'joomla':

config/auth.php — секция providers
'providers' => [
    'users' => [
        'driver' => 'joomla',  // было: 'eloquent'
        'model'  => env('AUTH_MODEL', App\Models\User::class),
    ],
],

Это единственная точка конфигурации. Всё остальное — guards, passwords — остаётся без изменений. Наш провайдер прозрачно заменяет стандартный.

Маппинг полей при миграции

При переносе пользователя в таблицу users переносятся только те поля, которые имеют прямой аналог. Остальные поля останутся пустыми — пользователь может заполнить их сам в личном кабинете.

Поле в pc0ec_users (Joomla) Поле в users (Laravel) Преобразование
email email Без изменений
username / name username username, если есть; иначе name
пароль (MD5 или bcrypt) password Перехэшируется в bcrypt через каст 'hashed'
block (0/1) published (1/0) Инвертируется: block=0 → published=1
registerDate created_at Если пусто — подставляется текущее время
ℹ️

Каст 'password' => 'hashed' в модели User автоматически применяет Hash::make() при записи через setAttribute(). Поэтому мы передаём в User::create() открытый пароль, а не хэш — Eloquent сам сделает bcrypt.

Мониторинг миграции

Поскольку каждый вошедший пользователь удаляется из pc0ec_users, в любой момент можно видеть, кто ещё не перешёл на новый сайт.

Просмотр через браузер

Страница /test выводит все оставшиеся записи из pc0ec_users в виде таблицы. Количество строк = количество пользователей, не заходивших на новый сайт.

SQL-запросы для статистики

-- Сколько пользователей ещё не мигрировали
SELECT COUNT(*) AS remaining FROM pc0ec_users;

-- Сколько уже мигрировало
SELECT COUNT(*) AS migrated FROM users;

-- Список не мигрировавших с датой регистрации
SELECT email, username, registerDate
FROM pc0ec_users
ORDER BY registerDate DESC;
✅

Когда таблица pc0ec_users опустеет — миграция завершена. На этом этапе можно вернуть в config/auth.php стандартный 'driver' => 'eloquent' и удалить файл app/Auth/JoomlaUserProvider.php как более не нужный.

Когда миграция завершена — как отключить провайдер

  1. В config/auth.php вернуть 'driver' => 'eloquent'
  2. Удалить строку Auth::provider('joomla', ...) из AppServiceProvider
  3. Удалить файл app/Auth/JoomlaUserProvider.php
  4. Удалить таблицу pc0ec_users из базы данных (убедившись, что она пуста)
  5. Выполнить php artisan config:clear
